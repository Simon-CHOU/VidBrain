# VidBrain 自演化知识库差距分析

## 一、项目目标定义

**理想状态：一个 Self-Evolving（自演化）的知识库**，应具备以下核心特性：

| 特性 | 说明 |
|:---|:---|
| **持续吸收** | 新素材自动流入，无需人工干预 |
| **自我优化** | 知识库质量随时间自动提升（链接更密、结构更清、内容更准） |
| **反馈闭环** | 能感知用户行为/编辑/纠正并反向调整自身行为 |
| **质量自知** | 能评估自身产出质量，识别并修复低质量笔记 |
| **增量演进** | 新知识能补充、修正或深化旧笔记，而非仅追加 |
| **结构自组织** | 知识图谱、分类、索引能随内容增长自动重组 |

| 能力维度 | 当前状态 | 目标状态 | 成熟度 |
|:---|:---|:---|:---:|

| **Agent 知识提炼** | 2 节点 LangGraph：纠错分段 + 双链织网 | ✅ 达成 | 中 |

| **精炼器（refine）** | 孤立笔记检测 + 双链补充 + MOC 生成 | ⚠️ 手动触发 | 低 |

## 三、差距详细分析

### Gap 1：精炼器未自动化（P0 - 关键）

**现状**：`refiner.py` 已实现完整功能（双链补充 + MOC 生成），但仅能通过 `--refine` 手动触发。

**影响**：知识库不会随着笔记增多自动优化质量——孤立笔记永远孤立，MOC 不会更新。

**目标**：
- 在持续模式中，每处理 N 批视频或每 T 小时后自动触发一次 refine
- CLI 支持 `--auto-refine-every 24h` 或 `--refine-after 10`（处理10批后精炼）

---

### Gap 2：无法增量更新已有笔记（P0 - 关键）

**现状**：管线只写新笔记，refiner 只在笔记末尾追加 `[[链接]]`。术语纠错、内容修正、信息补充等永远不会触及已有笔记的正文。

**影响**：
- 早期笔记（用 tiny 模型转录的）术语错误永远存在
- 新视频出现更好的同类解释时，旧笔记不会获得引用或补充
- 知识库"自演化"的核心——内容随时间变好——缺失

**目标**：
- 新增一个 Agent 节点：`update_existing_node`，处理新视频时检查是否有相关旧笔记需要更新
- LLM 判断新旧内容关联度，决定是否在旧笔记中追加引用或补充说明
- 可选：对旧笔记的术语进行回溯修正（需记录修正历史）

---

### Gap 3：没有用户反馈闭环（P1 - 重要）

**现状**：用户在 Obsidian 中编辑笔记（修正术语、删除错误双链、重命名、添加内容），系统完全不知情。

**影响**：系统的"智能"不会随使用而增强——犯过的错误会重复犯。

**目标**：
- 扫描 vault 时对比 `status: auto-generated` + `created` 时间 vs 文件修改时间，检测用户编辑
- 从用户编辑中提取反馈信号：删除的双链 = 不该连的，新增的双链 = 应该连的
- 将反馈用于优化后续笔记的 Agent prompt（如"用户倾向于连接 A 和 B，下次优先考虑"）

---

### Gap 4：没有质量评分 / 置信度（P1 - 重要）

**现状**：所有笔记在系统中地位平等，无法区分高质量笔记和低质量笔记。

**影响**：
- 无法优先修复最需改进的笔记
- Agent 在生成双链时不知道哪些笔记质量更高（应优先链接高质量笔记）
- 无法量化"知识库在变好"

**目标**：
- 为每篇笔记维护一个 `quality_score`（在 front-matter 或 DB 中）
- 评分维度：ASR 文本长度/段数、Agent 输出的结构化程度、双链密度、用户是否曾编辑过
- 在 vault 扫描和 Agent 处理中利用评分（如高质量笔记权重更高）

---

### Gap 5：失败任务无自愈机制（P1 - 重要）

**现状**：`process_pipeline` 抛异常 → `FAILED` 状态 → 停止，永不重试。

**影响**：API 临时波动、网络瞬断、偶发 OOM 等可恢复错误也会导致视频永远不被处理。

**目标**：
- DB 新增 `retry_count` 和 `last_error` 字段
- 失败任务自动调度重试（指数退避，最多 3 次）
- 持续失败的任务标记 `PERMANENTLY_FAILED`，并输出诊断建议
- 用户可手动 `--retry-failed` 重新入队

---

### Gap 6：分类器太粗糙（P2 - 改进）

**现状**：纯关键词匹配，存在：
- 946 个 unclear 视频被放弃（可能包含技术内容）
- BV1 规则过于宽泛（可能误分类非技术视频）
- 关键词不覆盖新兴技术（如 Modula、Zig 等不被识别）

**目标**：
- 分类增加 LLM 兜底：unclear 视频可选地调用一次廉价 API 根据文件名语义分类
- 分类器支持学习：用户手动分类的反馈能调整规则权重
- 定期从 vault 中提取新术语补充白名单

---

### Gap 7：跨笔记内容关联太弱（P2 - 改进）

**现状**：Agent 仅基于笔记文件名（标题）做双链匹配，不读取笔记内容。

**影响**：
- 标题不包含关键术语但内容是讲该主题的笔记，永远不会被链接
- 双链精度依赖标题命名质量
- 无法发现"A 笔记第 3 段和 B 笔记第 5 段讲同一个概念"这种细粒度关联

**目标**：
- Agent 或 Refiner 在运行时读取笔记的前 N 字符或摘要而非仅看标题
- 可选：为每篇笔记维护一个 `summary` / `tags` 字段（由 LLM 生成，写入 front-matter），用于更精准的关联匹配

---

### Gap 8：知识去重与冲突检测（P2 - 改进）

**现状**：多个视频讲同一个主题时，各自生成独立笔记，没有合并或互引。

**影响**：知识库碎片化，同一概念的信息散落在多篇笔记中。

**目标**：
- Agent 处理新视频前，先检索 vault 中是否有高度相似的已有笔记（基于标题 + 摘要）
- 如有，可选择：a) 更新已有笔记补充新信息；b) 创建新笔记但添加双向链接

---

### Gap 9：MOC 无增量更新（P2 - 改进）

**现状**：`generate_moc_files` 检查文件是否存在，如果存在就跳过。

**影响**：第一批生成的 MOC 永远不会包含后续新增的相关笔记。

**目标**：
- MOC 文件支持增量更新：append 新发现的笔记到已有 MOC 的分类下
- 定期重新分析全库主题，如结构变化则重写 MOC

---

### Gap 10：缺少测试与质量保障（P3 - 基础）

**现状**：3 个单元测试文件共 19 个用例，无集成测试，无端到端测试。

**影响**：改代码时不敢确定是否破坏已有功能，新功能的正确性无法自动验证。

**目标**：
- 补充 `test_pipeline.py`（mock API 的集成测试）
- 补充 `test_agent_graph.py`（验证 Agent 输出格式）
- 建立冒烟测试套件：固定测试视频 → 验证输出笔记的结构正确性

---

## 四、实施优先级路线图

```
Phase 1 (快速见效): Gap 1 精炼自动化 + Gap 5 失败重试
    ↓  约 2 个模块改动，投入小，立即提升系统自治性

Phase 2 (核心能力): Gap 2 增量内容更新
    ↓  涉及新 Agent 节点 + 笔记扫描逻辑，是"自演化"的核心

Phase 3 (智能增强): Gap 3 用户反馈 + Gap 4 质量评分
    ↓  让系统能从使用中学习，开始真正的"演化"

Phase 4 (深度优化): Gap 6-9 分类/关联/去重/MOC 增强
    ↓  提升知识库的精度和结构质量

Phase 5 (工程基础): Gap 10 测试体系
    ↓  为所有以上修改提供安全网
```

---

## 五、Phase 1 详细方案（精炼自动化 + 失败重试）

### 5.1 精炼自动化

**改动文件**：`vidbrain/main.py`

**具体变更**：
1. 新增 CLI 参数 `--auto-refine-after`（整数，处理 N 批后自动精炼）和 `--auto-refine-every`（时间间隔）
2. 在持续模式的 while 循环中，维护批处理计数器
3. 达到阈值后自动调用 `run_refine(cfg)`
4. 在 `PipelineConfig` 中新增对应字段

**改动文件**：`vidbrain/config.py` — 新增 `auto_refine_after: int = 0`、`auto_refine_every_hours: int = 0`

### 5.2 失败重试机制

**改动文件**：`vidbrain/db.py`
- 在 `video_pipeline` 表新增 `retry_count INTEGER DEFAULT 0` 和 `last_error TEXT`
- 新增方法 `get_failed_retryable()` 获取可重试的失败任务（retry_count < 3）
- 新增方法 `increment_retry(video_id)` 自增重试次数

**改动文件**：`vidbrain/pipeline.py`
- `process_pipeline` 在 catch 块中调用 `db.increment_retry(video_id)` 而非直接标记 FAILED
- 仅当 retry_count >= 3 时标记 PERMANENTLY_FAILED

**改动文件**：`vidbrain/main.py`
- `process_batch` 在处理前先检查并处理可重试任务
- 新增 `--retry-failed` flag 手动重试所有失败任务

## 六、后续 Phase 预览

### Phase 2 核心：增量内容更新
- 新文件 `vidbrain/updater.py`：笔记更新器
- 新 Agent 节点：判断新视频内容与已有笔记的关联
- 更新策略：追加补充 → 术语修正 → 双向链接

### Phase 3 核心：反馈与评分
- 新文件 `vidbrain/feedback.py`：用户编辑检测 + 反馈信号提取
- 新字段 `quality_score` 在 DB 和 front-matter 中
- Agent prompt 动态注入反馈（从反馈中学习的术语/链接偏好）

## 七、决定性假设

1. **笔记本体不会被删除**：增量更新指追加/修正，不涉及删除笔记（安全策略）
2. **API 费用可接受**：自动化精炼和重试会增加 API 调用量，假设这在预算内
3. **DeepSeek API 能力足够**：所有 LLM 任务继续使用 `deepseek-v4-flash`
4. **Vault 目录持续可访问**：所有自动化操作依赖 vault 目录的读写权限
