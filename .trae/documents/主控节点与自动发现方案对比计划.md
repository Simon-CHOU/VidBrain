# 主控节点与自动发现方案对比计划

## Summary

基于当前仓库代码与既有研究文档，比较两套方案：

1. 你提出的方案：
   - `Laptop` clone 一份 `VidBrain`
   - 启动时通过参数声明“我是算力节点 server”
   - `Desktop` / 主机上的 `VidBrain` 启动时以参数声明“我是 agent 主进程”
   - 主进程启动成功后扫描局域网端口，自动匹配可用算力节点

2. 现有推荐方案：
   - `Desktop` 保持唯一主控进程
   - `Laptop` 仅承担远端 ASR worker
   - 通过稳定主机名或显式 endpoint 连接 worker
   - 通过健康检查、熔断、冷却和后台恢复决定是否使用远端 GPU

结合你的偏好：
- 优先目标：长期最稳
- 节点定位：接受手填主机名 / IP
- 使用阶段：可长期日用

最终结论：
- **若以长期稳定性为第一优先，现有推荐方案更好。**
- 你提出的“节点角色参数化”部分是值得吸收的。
- 你提出的“局域网扫描端口自动匹配”不应作为主路径，只适合作为后续可选增强。

## Current State Analysis

### 当前代码事实

以下结论来自仓库现状，而不是推测：

1. `src/cli.py`
   - 当前只支持 `--asr-backend cpu|vulkan`
   - 没有“主控进程 / 远端节点 / worker server / discover”相关 CLI 参数

2. `src/models/config.py`
   - `PipelineConfig` 仅包含单机运行所需字段
   - 没有远端节点 endpoint、健康检查、熔断、恢复探测等配置位

3. `src/main.py`
   - 启动流程默认在本机初始化完整主流程：数据库、分类、watcher、ASR 引擎、pipeline
   - 当前是单实例主控模型，依靠 `acquire_singleton()` 保证同机只启动一个主进程
   - 没有“worker 模式”入口，也没有节点发现逻辑

4. `docs/vidbrain-remote-asr-feasibility.md`
   - 已明确推荐：`Desktop -> HTTP -> Laptop whisper-server`
   - 已明确不推荐纯 `SSH`、共享目录轮询，以及把 `mDNS` 当唯一真相源
   - 已明确推荐：稳定主机名 endpoint + 主动健康检查 + 熔断冷却 + 后台恢复

### 当前推荐方案的本质

现有方案本质上是：

- **控制面保守**
  - 先假定节点地址是已知的
  - 用应用层健康检查判断“是否真的可用”

- **数据面简单**
  - 主机本地抽 WAV
  - 远端只做 ASR

- **回退路径清晰**
  - 远端失败就走本地 CPU
  - 不做运行中任务迁移

这套设计与当前代码结构高度兼容，因为 `src/main.py` 仍然只有一个主控入口，远端节点只需要新增一条 ASR 服务路径，而不需要把整套主流程拆成双角色系统。

### 你的方案的优点

你的方案有两个真实优点：

1. **部署心智统一**
   - 两台机器都跑同一个仓库
   - 通过启动参数决定身份
   - 对个人项目来说，操作很直观

2. **后续扩展潜力更高**
   - 如果未来不止一个算力节点，角色化启动方式比“单独维护 whisper-server”更统一

### 你的方案的主要问题

对“长期最稳、可长期日用”这个目标来说，问题主要出在“自动扫描端口匹配节点”这部分：

1. **扫描结果不等于节点 ready**
   - 端口通了，不代表模型已加载
   - 端口通了，也不代表 GPU 当前可接单

2. **Windows / 家庭局域网环境不稳定**
   - 休眠恢复、多网卡、网络类型切换、防火墙规则都可能导致扫描误判

3. **冷启动和恢复时序复杂**
   - 主控比节点先启动时，会扫描不到
   - 节点刚上线时，端口可能开了但模型还没 ready

4. **引入了不必要的主流程耦合**
   - 如果 `Laptop` clone 完整 `VidBrain`，但实际上只承担 ASR worker，那么数据库、watcher、vault、LLM 等主流程依赖在节点机上是多余复杂度

5. **对当前代码改造面更大**
   - 不是只新增远端 ASR 路径
   - 而是要把现有单入口流程拆成“主控模式”和“节点模式”

## Proposed Changes

### 结论性建议

如果现在进入实现阶段，建议采用 **折中后的主方案**：

1. 保留你的“角色参数化”思想
2. 放弃“自动扫描端口作为主发现机制”
3. 采用“显式节点地址 + 健康检查 + 回退”的主路径
4. 把“自动发现 / 扫描”放到第二阶段，作为可选增强

### 推荐方案排序

#### 方案 A：现有推荐方案，稍微吸收你的角色化思想

推荐度：**最高**

做法：
- `Desktop` 仍是唯一主控
- `Laptop` 只跑 ASR worker
- `Desktop` 通过 `--remote-asr-host` 或类似参数连接 worker
- 可增加 `--run-asr-worker` 这类模式，让仓库本身也能启动 worker，但 worker 只启动 ASR 服务，不启动完整主流程

为什么更好：
- 最稳
- 与当前代码最兼容
- 日常维护成本最低

#### 方案 B：完整角色化 + 显式地址，不做自动扫描

推荐度：**中高**

做法：
- 两边都 clone `VidBrain`
- `Laptop` 用 `--role worker`
- `Desktop` 用 `--role primary --worker-host <hostname>`
- 不扫描局域网，只直连显式地址

为什么可接受：
- 保留你想要的统一部署体验
- 避免自动发现的不稳定

为什么仍次于方案 A：
- 需要把当前主流程拆出 worker 启动模式
- 改造量比“直接接 whisper-server / 远端 ASR 服务”更大

#### 方案 C：完整角色化 + 自动扫描端口匹配节点

推荐度：**最低**

做法：
- 两边 clone 同仓库
- 主控启动后扫描网段或常见端口
- 自动匹配声称自己是 worker 的节点

为什么不推荐：
- 对你的长期稳定目标不合适
- 最容易在 Windows 家庭网络环境里出现“发现到了但其实不可用”的假阳性
- 代码复杂度、测试复杂度、故障排查成本都最高

### 实施策略建议

按照你的目标，推荐按两阶段推进：

#### 第一阶段

- 不做自动扫描
- 只做：
  - 节点角色参数
  - 主控角色参数
  - 显式 endpoint
  - 健康检查
  - 熔断 / 冷却 / 恢复

#### 第二阶段

- 只有在你确认“地址维护真的很痛苦”之后
- 才加：
  - 可选自动发现
  - 但自动发现只负责“提供候选地址”
  - 真正切换仍由健康检查决定

### Windows Task Scheduler 的 worker 自启动示例

如果你希望 `Laptop` 开机后自动把自己拉起成算力节点，推荐在 `Task Scheduler` 里把 `VidBrain worker` 配成开机自启动任务。

#### 推荐命令

下面是直接对应当前实现的 `worker` 启动命令：

```powershell
uv run python -m vidbrain.main --role worker --asr-backend vulkan --model-size tiny --remote-asr-port 8080
```

如果这台 `Laptop` 还没配置好 `vulkan`，可以先退回 CPU 版本：

```powershell
uv run python -m vidbrain.main --role worker --asr-backend cpu --model-size tiny --remote-asr-port 8080
```

#### Task Scheduler 填写示例

更稳妥的方式是让 `Task Scheduler` 调用 `powershell.exe`，并在参数里先切到仓库目录再启动 worker。

`Program/script`

```text
powershell.exe
```

`Add arguments`

```text
-NoProfile -ExecutionPolicy Bypass -Command "Set-Location 'F:\ML\bailian-playground\VidBrain'; uv run python -m vidbrain.main --role worker --asr-backend vulkan --model-size tiny --remote-asr-port 8080"
```

`Start in`

```text
F:\ML\bailian-playground\VidBrain
```

#### 建议的 Task Scheduler 选项

- Trigger: `At log on` 或 `At startup`
- 勾选 `Run whether user is logged on or not`
- 勾选 `Run with highest privileges`
- 在 `Conditions` 里关闭“只有接通交流电才运行”之类会影响 Laptop 稳定性的限制
- 在 `Settings` 里启用“如果任务失败则自动重试”

#### 说明

- 这只是 `worker` 自启动，不会把 `Laptop` 变成主控机
- `Desktop` 仍然通过显式 `--remote-asr-host <Laptop主机名>` 连接它
- 这套方式仍然符合本计划的核心结论：`显式地址 + 健康检查` 优先，**不是**自动扫描发现

## Assumptions & Decisions

### 已锁定前提

1. 目标不是最快 PoC，而是长期日用稳定性
2. 你接受手填主机名 / IP
3. 当前仓库仍是单主控架构，不存在现成 worker 模式
4. 当前远端 ASR 文档已经明确倾向“显式 endpoint + 健康检查”

### 本次比较结论

1. **若只问“哪个更稳”**
   - 现有方案更好

2. **若只问“哪个看起来更省心”**
   - 你的“同仓库 + 角色参数”更直观

3. **若问“长期日用应该选哪个”**
   - 选现有方案为主
   - 吸收你的角色参数化思想
   - 不把自动扫描作为第一阶段或默认方案

4. **推荐的一句话版本**
   - **主控/节点角色化可以要，但节点发现应以显式地址 + 健康检查为主，自动扫描最多做辅助。**
