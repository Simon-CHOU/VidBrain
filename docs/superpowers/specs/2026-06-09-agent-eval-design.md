# Agent EVAL 评测系统设计文档

## 目标

通过全自动的 A/B 盲评，量化对比 main 分支 Agent（无 RAG）与 embedding-branch Agent（含 RAG 增强）的输出质量，判定 embedding-branch 是否可以合并。

## 整体架构

```
┌──────────────┐     ┌──────────────┐
│ main 分支     │     │ emb 分支      │
│ 完整管道      │     │ 完整管道      │
│ ASR→Agent→写  │     │ ASR→RAG→Agent→写│
└──────┬───────┘     └──────┬───────┘
       │                    │
       ▼                    ▼
  vault_main/          vault_emb/
  (笔记集 M)           (笔记集 E)
       │                    │
       └────────┬───────────┘
                │
                ▼
       ┌────────────────┐
       │  LLM Reviewer   │
       │  逐对盲评        │
       │  不知道M/E身份   │
       └───────┬────────┘
               │
               ▼
       ┌────────────────┐
       │  汇总报告        │
       │  4维度得分 + 结论 │
       └────────────────┘
```

## 核心决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 评分者 | LLM-as-judge 全自动 | 可批量跑，prompt engineering 消除偏误 |
| 测试数据 | 全新视频完整管道对跑 | 最真实可靠 |
| eval 架构 | 离线批量对跑，盲评 | 自动化 + 结果可复现 |
| 视频数量 | 20-30 个 | 足够统计显著性，可一天跑完 |
| 主题覆盖 | AI/ML、系统/性能、编程语言、架构设计 | 各 1/4 |
| 难度分层 | 新领域 50% + 已有领域 50% | 测 RAG 在有无参考时的表现差异 |

## 评分体系

### 分值设计

每个维度 1-5 分：

| 分 | 含义 |
|----|------|
| 1 | 明显差——有事实错误或严重遗漏 |
| 2 | 偏弱——方向对但质量低 |
| 3 | 及格——能用，无明显缺陷 |
| 4 | 好——准确、有用 |
| 5 | 优秀——精准、有洞察力 |

强制规则：每个维度必须先引用笔记原文作为证据，再打分。

### 维度 A：术语纠错准确度

对照 ASR 原始文本中的术语错误，评估每篇笔记的纠正质量。

```
评分锚点：
1 = 原本对的术语被改错，或明显错误完全没纠正
3 = 主要错误都纠正了，未引入新错误
5 = 不仅纠正了错误，术语写法与常见技术文档风格一致

证据要求：至少引用 2 处术语处理的具体例子
```

### 维度 B：双链相关性

检查笔记中每个 `[[wikilink]]` 是否合理，以及是否有应链未链。

```
评分锚点：
1 = 大量错误链接或严重漏链
3 = 大部分链接合理，无重要遗漏
5 = 每个链接精准命中，不漏不滥，密度恰到好处

证据要求：至少列举 2 个具体链接决策并判断对/错/漏
```

### 维度 C：更新建议质量

评估 `suggest_update` 节点输出的每一条建议。

```
评分锚点：
1 = 建议全部不相关或内容空洞
3 = 大部分建议合理，内容有一定价值
5 = 每条建议都精准针对目标笔记的知识缺口

特殊规则：如果两边都没有更新建议，此维度打 3 分（中性）
```

### 维度 D：综合质量

在完成 A/B/C 独立评分后，做整体判断。

```
综合考虑知识提炼深度、段落结构逻辑性、可读性与实用性。
输出：倾向 A / 倾向 B / 持平 + 理由（≤50 字）
长不等于好——简洁而准确 优于 冗长而空洞
```

## Reviewer Prompt 设计

### System Prompt（角色设定 + 评分规则）

```
你是一个技术文档质量评审专家。你的任务是对比两篇由 AI 生成的
技术笔记，从四个维度评判哪一篇更好。

## 身份规则
- 两篇笔记被标记为"笔记 A"和"笔记 B"。你不知道它们分别来自哪个系统。
- 你的评判必须基于证据，不能猜测来源。

## 评分规则
对每个维度独立打分（1-5 分），然后做综合判断。

### 维度 A：术语纠错准确度
- 首先，列出 ASR 文本中你认为存在错误或可疑的技术术语（至少 2 处）
- 然后，检查每篇笔记如何纠正这些术语
- 评分：1=改错或漏改, 3=主要错误纠正, 5=术语风格一致
- 必须引用笔记中的原文片段作为证据

### 维度 B：双链相关性
- 检查每个链接：上下文确实提到？漏链？过度链接？
- 评分：1=大量错误/漏链, 3=大部分合理, 5=精准命中
- 必须列举至少 2 个具体的链接决策并判断对/错/漏

### 维度 C：更新建议质量
- 检查每条建议：目标是否相关？类型是否合理？内容是否言之有物？
- 如果两边都没有更新建议，此维度打 3 分（中性）
- 评分：1=全部不相关, 3=大部分合理, 5=精准针对知识缺口

### 维度 D：综合质量
- 在完成 A/B/C 独立评分后，做整体判断
- 考虑：知识提炼深度、段落结构逻辑性、可读性与实用性
- 输出：倾向 A / 倾向 B / 持平，理由 ≤50 字
- 长不等于好。简洁而准确 优于 冗长而空洞

## 输出格式
必须输出严格 JSON，不得包含任何其他文字。
```

### User Message 模板

```
## ASR 原始文本
{raw_text}

---

## 关联笔记列表（两边 Agent 共用）
{related_notes_summary}

---

## 笔记 A
### 完整内容
{note_a_content}

### 双链列表
{note_a_links}

### 更新建议
{note_a_suggestions}

---

## 笔记 B
### 完整内容
{note_b_content}

### 双链列表
{note_b_links}

### 更新建议
{note_b_suggestions}

---

请按照评分规则对比这两篇笔记，输出 JSON。
```

### JSON 输出结构

```json
{
  "asr_issues": ["术语问题1", "术语问题2"],
  "score_a": {
    "笔记A": {"分": 3, "证据": ["原文片段"]},
    "笔记B": {"分": 4, "证据": ["原文片段"]}
  },
  "score_b": {
    "笔记A": {"分": 3, "证据": [{"链接": "[[CUDA]]", "判断": "对", "理由": "..."}]},
    "笔记B": {"分": 4, "证据": [{"链接": "[[GPU]]", "判断": "漏", "理由": "..."}]}
  },
  "score_c": {
    "笔记A": {"分": 3, "证据": ["评估说明"]},
    "笔记B": {"分": 3, "证据": ["评估说明"]}
  },
  "score_d": {"偏好": "A", "理由": "不超过50字的理由"},
  "self_doubt": {"最可能出偏误的维度": "B", "原因": "不超过50字"}
}
```

### 反偏误措施

1. **身份盲化**：只标记 `笔记 A` 和 `笔记 B`，不透露分支信息
2. **先举证后打分**：每个维度必须先摘录原文证据再给分
3. **self_doubt**：打完分后追问最可能出偏误的维度和原因
4. **长度中立**：明确提示"长不等于好"
5. **A/B 随机化**：每对视频重新随机打乱 A/B 标签

## 评测脚本架构

### 文件布局

```
eval/
├── eval_runner.py          # 主脚本：批量对跑 + 调用 Reviewer
├── reviewer_prompt.py       # System prompt + User message 模板
├── aggregator.py            # 汇总 4 维度得分 + 统计结论
├── video_list.txt           # 待测视频路径列表（用户提供）
└── results/
    ├── main_output/         # main 分支的笔记输出
    ├── emb_output/          # embedding 分支的笔记输出
    ├── reviews/             # 每对的 Reviewer JSON 结果
    └── summary.json         # 汇总报告
```

### eval_runner.py 流程

```
1. 读取 video_list.txt
2. 对每个视频循环：
   a. 创建独立临时 vault（main_vault / emb_vault）
      —— 两者从同一个 seed vault 复制，初始状态完全相同
   b. main 分支管道处理 → 生成笔记 M
   c. emb 分支管道处理 → 生成笔记 E
   d. 提取两边的 final_markdown、wikilinks、update_suggestions
   e. 调用 LLM Reviewer 盲评 → 保存 JSON
   f. 清理临时 vault
3. 全部完成后 aggregator.py 汇总统计
```

### 关键实现要点

- **独立 vault 隔离**：每个视频对使用全新临时 vault，防止 chunk 状态交叉污染
- **seed vault**：用户指定初始 vault 目录（模拟真实知识库），两边从相同的 seed 启动
- **顺序随机化**：每个视频对随机决定先跑 main 还是先跑 emb

### 汇总统计逻辑

```
对每一对视频：
  维度 A 分差 = 笔记E得分 - 笔记M得分  (+1/0/-1)
  维度 B 分差 = 笔记E得分 - 笔记M得分  (+1/0/-1)
  维度 C 分差 = 笔记E得分 - 笔记M得分  (+1/0/-1)
  维度 D 结果 = E赢 / 持平 / M赢

汇总输出：
  - 每个维度的平均分差（正 = embedding 赢）
  - embedding 综合胜出比例（D维）
  - 按新领域 vs 已有领域分层统计
  - self_doubt 加权：某维度被怀疑偏误 > 30% → 标注低置信
```

### 结论判定规则

```
embedding 被认为"更好"的充分条件：
1. A+B+C+D 综合胜出比例 >= 60%
2. D 维 embedding 胜出比例 >= 55%
3. 已有领域子集的 D 维胜出比例 >= 新领域子集
   （证明确实是 RAG 在增益，而非随机波动）

如果 3 不满足但 1+2 满足 → embedding 整体略好但 RAG 增益不明显
如果 1 或 2 不满足 → 不合并，继续打磨
```

## 边界情况 & 错误处理

| 情况 | 处理 |
|------|------|
| 视频 ASR 失败（音频损坏/静音） | 跳过该视频，记录到 `results/skipped.json` |
| Agent 调用失败（API 超时） | 重试 1 次，仍失败则跳过该视频 |
| 某篇笔记无双链 | 维度 B 正常评分，证据栏标注"无双链" |
| 某篇笔记无更新建议 | 维度 C 打 3 分（中性），证据栏标注"无更新建议" |
| Reviewer 输出非合法 JSON | 重试 1 次，temperature 降到 0；仍失败则标记"评审失败" |
| Reviewer self_doubt 反复指向同一维度 | 汇总时该维度权重降为 0.7x |
| 新领域视频 vault 无相关笔记 | 正常——这正是测试分层的目的 |
| seed vault 为空 | 仅测"新领域"场景，报告需标注此限制 |
| 两边输出完全一样 | 标记为"无差异"，纳入统计但不计入胜负 |

## 不做

- 不修改 main 分支或 embedding-branch 的任何生产代码
- 不引入新的依赖（复用现有 openai、numpy 等）
- eval 脚本独立运行，不侵入管道
